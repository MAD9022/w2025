# 1. SPA Assignment - Part One

This first part of the Single Page Application will be focused on the navigation and data caching for a single page with three sections - a home page, a list (users) page, and a details (user) page.

The URLs for the app will be:

  1. /# for the home screen (or /index.html#)
  2. /#users for the list screen (or /index.html#users)
  3. /#details for the details screen (or /index.html#details)

The index.html file is the only HTML file in the web app.

At the top of every screen is the same navigation bar with links to the home screen and the list screen.

On the home screen there should be a welcome message and a button to refresh the data.

On the list (users) screen, in the second part, we will be creating a list or grid of user cards. In this first part we only need a title plus a few links that sends the user to /#details and pass the user id through the history.state object.

On the details (user) screen, in the second part, will we be displaying all the user data. In this first part we only need a title and the user id.

All three screens should, combined with the navbar, fill a minimum of the whole vertical height of the screen.

# Events

The app needs a DOMContentLoaded listener will call a function that will read the current url, get the hash value and show the appropriate screen:

  • # or null: will show to the home screen
  • #users: will show the list screen
  • #details: will show the details screen
  • any other hash value needs to trigger a call to replaceState to rewrite the hash value as # and then show the home screen.

The app needs a popstate listener that will handle the forward and back buttons and call a function to show the appropriate screen.

All three events should call the same function to change the UI display and show the appropriate screen.

The button on the home screen needs a click listener, which will make a fetch call to https://random-data-api.com/api/v2/users?size=12. If the fetch call returns a valid response, then save it with the Cache API. Use the URL for the fetch as the request for the cache.

Git Commits: There should be at least 5 commits in your repo.

# 2. SPA Assignment - Part Two

The premise for the site could be something like an admin site to view all the current users on the site and their details.

The second part of the SPA is adding the content from the data to the users list screen each time the user navigates to it. Each user card should be wrapped in an anchor tag that navigates to #details. The uuid from the user should be added to the querystring in the anchor tag href attribute. When the querystring changes, the page will reload.

As an alternative to the querystring and reloading of the page, you may use a click listener on the user card, call the pushState method and pass the user id value through the state object being passed to pushState. This will require you to also call the function to "navigate" to the details screen. By "navigate" we mean to hide the #users screen and show the #details screen.

When the page loads, check the Cache for the users json file, and load that into a global variable. If the data is not in the Cache when the page loads, then the #users screen should display a message about there being no users currently and a button, just like the one on the home screen to load latest users.

On the #users screen, display a responsive list or grid of cards. One card for each of the dozen users. Each card should display the first_name, last_name, username, email, and avatar of each user.

When the #users screen loads, check in the cache for user data, retrieve the json data from the cache, save the user array in a global variable, and build the user cards. Each user card needs to be wrapped in an anchor tag that is linked to the #details screen with a URL that looks like: index.html?uid=asdf-23423-sdfsd-asdf#details. The hash value is the page to display. The querystring is the id of the user to retrieve from the user array that was retrieved from the cache. Or, as previously mentioned, you can use the pushState method combined with the history.state property to pass the user id from #users to #details.

If there is NO user data in the Cache, or the user is clicking one of the buttons to load new data, then the sequence will be slightly different.

Fetch -> clone response -> one response into the cache -> one response uses .json() to get array of users

Once you have the user data from the cache/fetch, then you need to make sure that you're URL is #users and display the user cards.

If there is user data in the cache, then all you need to do is read the data and build the cards when the user navigates to #users.

On the home screen, when the user hits the button to fetch new user data you will be replacing the data in the cache with the new response. You will also need to send the user to the #users screen. This will require calling pushState to change the location and save it to the history stack, and then calling your function to update the UI and displayed screen.

The #details screen needs to display all the information from the user card plus at least 5 other fields from the user data.

The button for updating the user list by calling fetch and saving the response to the Cache is always available on the home screen. On the #users screen the button only appears when there is no data.

Avatar Images You should have a default image to use for user avatars in case the real avatar is missing or slow to load.

Google Font: You should use at least one Google font for your SPA. Include the 2 preconnect links and one css link in your HTML.

Content-Security-Policy: You should have a CSP meta tag in the head that limits the sources for all the files.

Git Commits: There should be at least 5 commits in your repo.

# Extra Features

Some other features that are not required for grades, but that give you something that you can do to enhance your application for a portfolio and for your own learning and practice.

  • Add a delete button on the #details screen that will remove the user object from the user array, and then create a new File + Response using that updated data. Then the response needs to be saved inside the Cache to replace the existing response.
  • Add a service worker that will cache all the HTML, CSS, JS, JSON, and font files. In this way the app can still work if offline. You will need to make an initial request for the user data when the SPA first loads, and is online. The buttons for fetching new users need to be made disabled when the page is offline. Listen for the offline event to achieve this.
